Building four AI agents
in two weeks.

How I designed, built, and shipped an agentic automation suite — from LinkedIn content publishing to a video-to-shorts pipeline — and what it taught me about AI product thinking.

Timeline2 weeks (MBA sprint)
RoleSolo builder — design, dev, PM
StackPython, LLM APIs, scheduling tools
Outcome10,000+ cross-platform impressions

The problem

The brief was open-ended: build something agentic. Most people in my MBA cohort interpreted that narrowly — one agent, one task, one API call. I wanted to understand what it actually feels like to design a system of agents, where each one has a defined scope, failure modes, and handoffs to the next.

I also had a real problem I wanted to solve: I was producing content and doing administrative work that followed predictable patterns. Predictable patterns are what agents are for.

"If I can describe the decision logic in a paragraph, a well-prompted agent can execute it. The interesting design question is: what breaks when the edge case arrives?"

The four agents

I scoped each agent to a single job with a clear success/failure state — the same way I'd write a user story for a PM project.

Agent 01
LinkedIn Content Generator & Publisher

Took a topic brief or raw notes and generated a post in my voice, formatted it for LinkedIn's algorithm (hook + body + CTA), and queued it for scheduled publishing. I defined the tone parameters based on my own writing — this was the most PM-intensive agent to design because "voice" is not a technical requirement, it's a product requirement.

Agent 02
Expense Claim Processor

Parsed receipt inputs, categorized expenses against a defined schema, flagged anomalies, and generated a formatted claim document ready for submission. Designed to handle ambiguity — e.g., a meal receipt that could be "team lunch" or "client entertainment" — and route edge cases for human review rather than guessing.

Agent 03
Office Hour Scheduling Agent

Managed availability parsing, conflict detection, and calendar invite generation. The interesting design decision here was what the agent should do when two preferences conflict — I deliberately built in a soft-decline message rather than silent dropping, because the failure mode matters as much as the success case.

Agent 04
Video-to-Short-Form Pipeline

The most technically complex agent. Took long-form video input, identified quotable or clip-worthy moments using transcript analysis, and output platform-ready short-form clips with suggested captions. This agent drove 10,000+ impressions across platforms — the clearest proof that the system worked at the output level.

What I learned

  • Designing an agent is a product exercise, not just an engineering one. The most important decisions are about scope boundaries and failure handling — exactly the same decisions you make when writing a PRD.
  • Voice is a product requirement. Agent 01 required more iteration than all others combined because "sounds like me" is an acceptance criterion that can't be unit-tested. I had to build my own evaluation rubric.
  • Edge cases are where agent trust is built or destroyed. I deliberately designed each agent to surface ambiguity rather than resolve it silently — that design principle came directly from my journalism instinct: never assume you have the full story.
  • The most valuable agent (by measurable impact) was the simplest one to describe. The video pipeline had one job: find the moments worth keeping. That specificity made it the easiest to evaluate and the most effective in practice.
